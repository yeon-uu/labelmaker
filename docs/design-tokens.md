## 대담함은 한 곳
출력 애니메이션(주르륵 + 완성 토스트)에만 화려함 허용. 나머지 UI는 조용하게. 그라데이션 남발·이모지 아이콘·카드 그림자 금지.

## 2026-07-04 (2차) — UI 전면 재구성 추가 토큰

### 기기 내장형 전환
컨트롤 패널(글리프/폰트/테이프/프레임/기호 카드) 전부 삭제. 모든 조작을 `.machine-body` 내부 absolute 배치로 이동(실물 라벨기 UX). 화면에는 기기 하나만 남는다.

### 키보드 컴팩트 치수
- 키 시각 크기: 36~40px (기존 44px 고정 대신, 좁은 키보드 패널에 3줄+하단줄 배치 위해 축소).
- 히트 영역은 44px 유지 — 시각 크기와 별개로 패딩/투명 확장 영역(`::before` 또는 padding)으로 44px 이상 확보.
- 키보드 grid는 패널(left 0.83% / top 32.30% / width 93.60% / height 36.94%, 몸체 기준) 안에 꽉 채운다.

### 기능키 줄 (MODE/방향키)
- 5개 소형 버튼(MODE, ▲▼◀▶), 키보드 패널 최상단 줄에 배치(LCD 아래 틈 공간 부족 판단).
- 크기: 시각 32px, 히트 44px(패딩 확장).

### 설정 모드 (state.mode)
- `state.mode`: `'text' | 'settings'`. `state.settingIndex`: 0~3 (폰트 → 테이프색 → 프레임 → 사이즈 순환).
- MODE 버튼으로 text ↔ settings 전환. ▲▼로 항목 이동(순환), ◀▶로 값 순환(순환).
- LCD 표시 형식(설정모드): `"{항목명}: {현재값} ◀▶"` 1줄.

### 라벨 사이즈 2종 (신규)
- **긴 스트립**(기존): width 280 / height 100 (표시 px, scale 전).
- **정사각**: width 160 / height 160. 여러 줄 텍스트 중앙 배치, `render.js`의 줄바꿈 로직이 maxWidth=160-padding 기준으로 동일하게 동작(정사각 폭이 좁아 자연히 더 많이 줄바꿈됨).
- 다운로드 PNG는 표시 px의 3배 해상도 유지.

### 라벨 모서리 radius 축소 (신규)
- 기존 `--radius-device: 16px`를 라벨 전용으로 쓰지 않고, 라벨 자체 radius는 `--radius-label: 3px`로 신설(거의 직각, 스트레이트 컷 느낌). 기기 본체 radius 개념(`--radius-device`)은 유지하되 라벨에는 미적용.
- `render.js drawLabel()`의 기본 `radius` 인자를 16 → 3으로 변경.

### 라벨 텍스처 (신규)
- SVG `feTurbulence` grain 오버레이, 불투명도 4%(3~6% 범위 중간값) — `--label-grain-opacity: 0.04`.
- 화면: `.label-canvas`에 CSS `filter` 또는 오버레이 `::after` + SVG filter로 근사, 혹은 canvas 자체에 노이즈를 합성(택1, 구현 시 확정해 아래에 갱신).
- Canvas PNG: 오프스크린 캔버스에 랜덤 그레이스케일 노이즈 픽셀을 만들어 `globalAlpha: 0.04`로 라벨 영역에만 합성 후 클리핑.

### 잉크 번짐 (신규)
- 텍스트에 `shadow-blur: 0.3px` 수준의 미세 blur만 추가(Canvas `ctx.shadowBlur`), 색은 잉크색과 동일(번짐이지 그림자 아님). 화면 미리보기는 LCD 텍스트에는 적용하지 않음(LCD는 디지털 표시라 인쇄 질감과 무관) — 오직 출력된 라벨(Canvas)에만.

### 완성 라벨 그림자
- `box-shadow: 0 2px 6px rgba(58,51,48,0.08)` 수준의 저채도 저투명도 1개만. 광택/그라데이션 금지(기존 규칙 재확인).

### 사운드 파라미터 (js/sound.js, 문서화 목적 — 실제 값은 코드가 원본)
- 키 입력음: sine/triangle 600~900Hz 랜덤(±10~15%), attack 5ms / release 40ms, 총 40~80ms.
- 스페이스/지우기: 400Hz대, 동일 엔벨로프.
- 출력 시작음: 100→200Hz linearRamp, 150ms.
- 인쇄 사운드: 화이트노이즈 버스트 짧게 게이팅, 6~8회 반복(감김 애니메이션과 동기화).
- 볼륨 기본 낮게, 음소거 상태 `localStorage['labelmaker:muted']`에 저장.
- AudioContext는 첫 사용자 상호작용 시 생성 + resume().

### 출력 모션 리듬 (신규)
- `--anim-print: 1400ms` 유지, 단일 ease 대신 6~8단계 `@keyframes`로 "감기며 올라옴"(멈칫 반복) + 마지막 2~3px 오버슈트.
- 슬롯 위치에 `overflow:hidden` 클리핑 컨테이너 신설(라벨이 슬롯 아래에서 보이지 않고 슬롯 틈에서만 나오는 것처럼).
- perspective + rotateX 2~4deg → 0deg 수렴(모서리 말림 느낌), 슬롯 위 옅은 접촉 그림자 1개.
- machine-wobble과 인쇄 사운드를 같은 6~8스텝 타이밍에 동기화.

### 라벨 텍스처: 실제 라벨 에셋 alpha 마스킹 (jimp 사용)
`assets/라벨지모음.png`에서 하단 중앙의 거의 정면 라벨 1장을 크롭 → `assets/label-texture.png` (525×205, RGBA PNG) 로 저장.
- 처리 도구: **Node.js `jimp`** (Python Pillow 미시도 — jimp로 충분히 처리 가능해 추가 설치 안 함). `npm install jimp`로 프로젝트 로컬 설치(`node_modules`, package.json 없음 — 원시 설치라 재현 시 `npm install jimp` 필요).
- 처리 과정: 원본에서 라벨 영역을 여백 포함 크롭 → 네 모서리 색을 샘플링해 bilinear 보간으로 로컬 배경 밝기(`bgLumAt(x,y)`) 추정(글로우가 코너마다 밝기가 달라 단일 threshold 대신 사용) → `alpha = clamp((lum - bgLum) / (labelLum - bgLum))^1.8`로 부드러운 알파 매핑 → 배경 완전 투명(alpha=0, 픽셀 스캔으로 4코너 확인), 라벨 안쪽은 alpha 255, 경계부는 원본의 얇은 회색 outline 라인이 자연스럽게 유지됨.
- 라벨 자체가 사진상 약 4도 정도 기울어져 있어 완전한 정면은 아니었음(회전 보정을 시도했으나 원본 해상도 한계로 완전한 deskive 실패, 최종적으로는 무보정 원본을 그대로 alpha 마스킹). 화면/PNG에서는 `background-size: cover`로 스트레치해 사용해 기울기가 시각적으로 거의 드러나지 않음.
- 화면 표시: `.label-canvas`의 배경이자 Canvas 렌더 시 `drawImage`로 텍스처를 라벨 크기에 맞춰 그린 뒤 그 위에 잉크색(#3A3330) 텍스트/글리프/프레임을 그림.
- 라벨 모서리는 기존 토큰대로 `--radius-label: 3px` 유지(텍스처 자체가 아니라 Canvas clip으로 라운드 처리).

### 실제 배열 vs 사용자 관찰 배열의 불일치 (정직하게 기록)
- **R2(한글) 자모 배열이 다름**: 사용자 관찰은 "ㅂㅈㄷㄱㅅㅛㅕㅑㅐㅔ"(10자모)였으나, 실측 이미지에는 **ㅔ 키가 존재하지 않는다** — 실제로는 `ㅂ ㅈ ㄷ ㄱ ㅅ ㅛ ㅕ ㅑ ㅐ [이모티콘]`(9자모+이모티콘=10칸). ㅐ와 ㅔ를 여러 차례 확대 스캔해 재확인했지만 ㅔ는 어느 열에도 없다.
- 이 때문에 Shift(쌍자음) 규칙 중 "ㅔ→ㅖ"는 이 물리 키보드로 구현 불가(누를 ㅔ 키 자체가 없음). ㅐ→ㅒ만 지원한다.
- R3(한글: ㅁㄴㅇㄹㅎㅗㅓㅏㅣ+BS), R4(한글: Shift ㅋㅌㅊㅍㅠㅜㅡ,.), R1, R5 구조는 사용자 관찰과 정확히 일치.
- **영문판 그리드 셀 수가 한글판과 다름**: 한글판은 R2/R3/R4 전부 10칸이지만, 영문판은 R2(QWERTYUIOP+Symbol)=11칸, R3(ASDFGHJKL+BS)=10칸, R4(Shift+ZXCVBNM+—,.)=11칸이다. "두 이미지가 완전히 같은 그리드를 공유해 레이어만 바뀐다"는 전제와 달리, 실제로는 언어별로 별도 좌표맵을 관리해야 했다(`js/app.js`에 R2/R3/R4를 KR/EN 각각 별도 상수로 정의).
- 영문 R3의 D/F 경계, L/BS 경계는 그림자 색 차이가 매우 미세해 픽셀 스캔으로 명확히 못 갈랐다 — 근접 열 폭 패턴(약 84~96px)에 맞춘 균등분할로 보정했다(오차 가능성 있음, 투명 히트존이라 실사용 영향은 적을 것으로 추정).

### LCD 픽셀 폰트
`DotGothic16`(Google Fonts, 한글 지원 픽셀 폰트)를 `index.html`의 `<link>`에 Jua/Gaegu와 함께 추가. **LCD 전용** — 출력 라벨(Canvas 렌더)은 기존 Jua/Gaegu/고딕 그대로, 절대 혼용하지 않음. LCD에 깜빡이는 커서(`.lcd-cursor`, 1s step-end 점멸)를 텍스트 모드에서만 표시.

### 키 누름 피드백
`.keyzone::after`가 기본 투명(`rgba(58,51,48,0)`)이고, `.is-pressed` 클래스가 붙으면 반투명 어두운 라운드 오버레이(`rgba(58,51,48,0.22)`) + `scale(0.96)`으로 90ms 표시 후 자동 해제(JS `setTimeout`, CSS 트랜지션 아님 — 순간 최대 강도로 표시했다가 트랜지션으로 사라지는 방식). 기능키(스페이스/BS/한영/기호토글 등)는 `playKeyLow()`로 더 낮은 톤 사운드, 숫자/자모/알파벳은 `playKey()` 기본 톤.

### 설정 메뉴 (환경 오벌) / 출력확인 플로우 (핑크버튼)
기존 2026-07-05(2차) 토큰의 `state.mode`(text/settings) 구조를 유지하되 `'confirm'` 모드를 신설. 핑크 버튼 또는 "인쇄" 오벌을 누르면 `state.mode = 'confirm'`으로 전환, LCD에 "출력하시겠습니까?" + "▸예 아니오"(좌우 화살표로 토글) 표시. 선택줄변경(Enter) 또는 핑크버튼 재클릭으로 확정 — 예=출력 진행, 아니오=입력모드 복귀(취소).

### 기호 오버레이 패널 (신규 CSS)
사진에 없는 계층이라 `.symbol-overlay`(반투명 크림 배경, `rgba(255,251,242,0.96)`, radius `--sp-12`, `box-shadow: 0 2px 10px rgba(58,51,48,0.12)`)를 키보드 영역 위에 절대배치로 덮는다. `.symbol-grid`는 6열 그리드, 유니코드 기호 20종 + 글리프 SVG 10종 = 30개, 콤보 프리셋 없음(개별 삽입만). 이 grid/overlay 색상·radius·그림자 값은 기존 카드 그림자 금지 규칙의 예외로 새로 추가한 토큰이다(패널 자체가 사진 위에 뜨는 임시 UI라는 성격상 최소한의 구분 그림자 허용).

### 375px 히트존 한계 (불가피, 정직하게 기록)

## 2026-07-05(4차) — 근본 재작업: "몸체 트림 이미지 + 오버레이" 구조

### 문제의 정확한 원인 (3차까지의 좌표가 왜 전부 어긋났는가)
3차까지는 `.machine-body`의 `background-image`에 **원본 이미지(투명/크림 여백 포함, 1536×1024)**를 그대로 썼고, 좌표는 "몸체 바운딩박스 기준 %"(예: x216~1311 → 폭 1095)로 계산했다. 그런데 CSS는 `background-size: 100% 100%`로 **원본 이미지 전체**를 `.machine-body`(몸체 바운딩박스와 같은 비율로 설정된 박스)에 늘려 그렸으므로, 오버레이 좌표(몸체 기준 %)와 실제 렌더링(원본 기준 %)이 서로 다른 좌표계였다 — 여백 비율만큼 모든 히트존·LCD가 어긋난 근본 원인. 이번 세션에서 이 구조 자체를 폐기했다.

### 1. 이미지 트림 (Python PIL)
`assets/머신한글키보드.png`(1536×1024), `assets/머신영어02.png`(1536×1024) 각각에서 배경 크림색(약 rgb(253.75,247,239))과의 RGB 거리(threshold 12)로 몸체 바운딩박스를 잡아 크롭:
- `kr_trim.png`: bbox (217,122)-(1312,913) → **1096×792**
- `en_trim.png`: bbox (217,122)-(1311,911) → **1095×790**

두 트림 이미지의 크기 차이(1px, 2px)는 트림 threshold의 미세한 차이일 뿐이며, 오벌·행 위치를 %로 비교하면 사실상 동일 챠시임을 재확인했다(크롭 비교 이미지로 육안 확인, 아래 각 좌표의 kr/en 값이 1% 이내로 근접).
